# 回归测试计划 - 集成测试

目标：在现有 Kubernetes 部署上执行端到端的集成回归测试，验证各组件在变更后的协同工作是否仍然可靠，覆盖 API、任务队列、文档转换服务和前端交互的关键流程。

范围：覆盖 /k8s 中的服务及其交互：FastAPI API、Gotenberg 文档转换服务、Celery 任务队列、Redis、Postgres、Nginx、以及相关前端交互与接口。

前提与环境：使用项目根目录下的 Kubernetes 配置（/k8s），在同一集群或等效测试集群中执行回归用例。确保测试数据和依赖服务可访问，并具备回滚方案。

1. 测试对象映射
- API 层：健康检查、可用格式、上传、转换、任务查询、PDF 合并/分割/提取等入口。
- 转换流程：从上传到下载的端到端流程。
- 任务队列与执行：Celery 作业的提交、执行、状态跟踪与完成。
- 外部服务：Gotenberg 的文档转换结果、Redis 队列状态、Postgres 持久化记录（若涉及）
- 前端（如果有）交互与显示的一致性。

2. 测试用例矩阵（选取部分典型用例，覆盖关键路径）
- IR-001: 完整转换端到端流程
  - 步骤：上传 Word 文件 → 转换为 PDF → 下载 PDF
  - 期望：任务成功，下载可用。
- IR-002: 批量转换端到端流程
  - 步骤：上传多文件 → 批量转换 → 全部下载确认
  - 期望：所有任务完成并可下载。
- IR-003: PDF 转换为 Word
  - 步骤：上传 PDF → 转换为 Word → 下载 Word
  - 期望：Word 文件可用且无损。
- IR-004: 长尾路径与错误处理
  - 步骤：提交无效输入（如不支持的格式）→ 检查错误响应
  - 期望：返回明确错误信息且状态码合理。
- IR-005: 并发压力测试
  - 步骤：发起 10 个并发请求（同一或多种格式）→ 监控系统稳定性
  - 期望：无崩溃，任务均进入正确状态。
- IR-006: 服务重启后的任务恢复
  - 步骤：中途重启一个或多个服务（如 Redis、Celery、Gotenberg）→ 继续查询任务状态
  - 期望：任务状态正确更新，支持断点续传。
- IR-007: 安全性与访问控制（如有鉴权）
  - 步骤：无授权访问尝试 → 应拒绝并给出正确提示
  - 期望：安全策略生效。

3. 测试数据
- 使用测试环境中准备的样例文件集：Word、PDF、Excel、PPTX、PNG、Markdown 等。
- 数据源：来自 6. 测试数据中的清单，确保大小和格式符合实际使用情境。

4. 执行流程与产出
- 步骤：在可控集群上执行测试脚本/用例，记录每个用例的执行时间、状态、日志与下载结果。
- 产出物：回归测试报告、每个用例的输入/输出、失败用例的诊断信息、截图/日志证据。

5. 成功标准
- 所有选定用例通过且符合预期输出。
- 日志可追溯，性能指标达到设定阈值（如并发、响应时间区间）。
- 无不可恢复的失败，必要时能提供回滚或降级路径。

6. 风险与缓解
- 风险：集群资源紧张、网络延迟影响测试稳定性。
- 缓解：在空闲资源时间窗口执行、并发度做自适应限制、记录详细日志。

7. 与现有测试的关系
- 与 4. 灰度测试框架协同，作为回归层面的关键集成场景。
- 与 Entropy Guardian 的周期性清理和健康检查策略相兼容，确保测试产出不会污染项目结构。

8. 产出与交付
- 回归测试报告（含汇总、逐项用例、问题清单、改进点）。
- 与 handoff 文档的对齐点：为后续 Agent 对接提供清晰上下文。
